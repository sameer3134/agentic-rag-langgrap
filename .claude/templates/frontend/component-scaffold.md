# Component Scaffold Generator

Generate a production-ready, fully-typed, accessible frontend component with all supporting files based on the Self-Hydrating system's conventions.

## Arguments

$ARGUMENTS — A natural-language description of the component to scaffold. Should include the component name, its type, and a brief description of its purpose. Example: `"UserProfileCard — a data display component that shows avatar, name, role, and status badge"`

---

## Step 1: Parse Component Requirements

From $ARGUMENTS, determine:

1. **Component Name** — PascalCase for the component, conforming to {{NAMING_CONVENTION}}
2. **Component Type** — one of the following:
   - `page` — A routable view that corresponds to a URL path
   - `layout` — A structural wrapper providing consistent chrome (nav, sidebar, footer)
   - `feature` — A self-contained business-logic unit composed of multiple UI primitives
   - `ui-primitive` — A low-level, highly-reusable presentational building block
   - `form` — An input-gathering component with validation and submission logic
   - `modal` — An overlay dialog with focus trapping and dismissal behavior
   - `data-display` — A read-only component for rendering structured data (tables, cards, lists)
3. **Description** — What the component does and why it exists
4. **Data Dependencies** — What props, API calls, or store slices the component needs

If the type is ambiguous, ask the user before proceeding. Never guess.

## Step 8: Accessibility Checklist

Before marking this component as complete, verify every item:

- [ ] **ARIA role** — Component uses the correct WAI-ARIA role for its type
- [ ] **ARIA label** — `aria-label` or `aria-labelledby` is present on the root interactive element
- [ ] **ARIA states** — Dynamic states (`aria-expanded`, `aria-selected`, `aria-checked`, `aria-disabled`) update correctly
- [ ] **Keyboard navigation** — All interactive elements reachable via Tab; activation via Enter/Space
- [ ] **Focus management** — Focus is trapped inside modals; focus returns to trigger on close
- [ ] **Focus visibility** — `:focus-visible` styles are distinct and meet 3:1 contrast ratio
- [ ] **Color contrast** — All text meets WCAG 2.1 AA (4.5:1 normal text, 3:1 large text)
- [ ] **Screen reader** — Tested with at least one screen reader (VoiceOver, NVDA, or JAWS)
- [ ] **Reduced motion** — Animations respect `prefers-reduced-motion: reduce`
- [ ] **Zoom** — Component is usable at 200% browser zoom without horizontal scroll
- [ ] **Error identification** — Form errors are programmatically associated with their fields via `aria-describedby`

Run automated a11y audit: `{{FE_A11Y_TOOL}} <ComponentName>`
